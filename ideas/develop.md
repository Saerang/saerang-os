# develop — 기획 → PR 개발 오케스트레이터

> 2026-06-10 인터뷰로 정리. 모드: 표준 (7문답)

## 한 줄 정의

Jira 티켓 하나(=PR 하나)를 받아 대상 프로젝트에서 구현 → 테스트 → 코드리뷰 에이전트 검증 → PR 생성까지 수행하는 개발 오케스트레이터.

## 배경 / 문제

개발리더로서 기획이 나온 뒤의 개발 진행을 AI에 위임할 구조가 필요하다.
"어떤 코드를 어떻게 올릴지"가 매번 수작업 — 티켓 단위로 반복되는 구현·검증·PR 흐름을
오케스트레이터로 체계화한다. (ClaudeOS 강의 Day1 오케스트레이터 + Day2 서브 에이전트 과제 반영)

## 대상

개발리더 본인. 기획을 작은 단위 Jira 티켓으로 쪼개는 것까지는 사람이 하고,
티켓 이후의 구현~PR을 오케스트레이터에 위임한다.

## 핵심 내용

**포함:**
- 입력: Jira 티켓 키 (기획 원문은 Jira/Confluence/md, 때로 Figma)
- 단위: 티켓 하나 = PR 하나
- 티켓 분석은 ticket-analyst 서브 에이전트가 수행 — "보강 필요"면 질문 목록만
  보고하고 종료 (fail-fast 티켓 품질 게이트, 보강 후 재실행)
- "진행 가능"이면 스펙 요약 확인 후 시작 승인 게이트 1회 → 이후 무인 진행
- check 모드: 여러 티켓을 병렬 분석만 해서 사전 스크리닝 (구현 없음)
- 대상 프로젝트는 OS 외부 경로 — 티켓 라벨 등으로 경로 매핑, 시작 시 지정 가능
- 프로젝트별 컨벤션 문서 로딩, 없으면 코드 분석으로 디폴트 생성 후 추가
- 검증: 테스트 통과 + 코드리뷰 서브 에이전트(pr-reviewer) 지적사항 반영
- 출구: PR 생성까지. 막히면 Draft PR + 상황 보고
- 작업량에 따라 단일 진행 / git worktree 병렬을 스스로 판단

**제외:**
- 기획 분해·Jira 티켓 생성 (사람 몫)
- PR 머지 판단, 배포 (사람 몫)

## 성공 기준

테스트와 리뷰를 통과한 코드가 정식 PR로 올라간 상태. 실패 시에도 Draft PR과
상황 보고(시도한 것 / 막힌 이유 / 선택지)가 남아 리더가 코드 수준에서 판단 가능해야 한다.

## 검토한 대안

- 출구를 "구현+로컬 검증까지"로 좁히기 → PR 생성까지 가는 쪽이 위임 효과가 커서 채택 안 함
- 분해까지 AI에 위임 → 작은 단위 티켓 분해는 사람이 직접 (품질 통제), AI는 PR 하나 기준만
- 실패 시 중단·보고만 → Draft PR을 함께 올려 코드 수준에서 판단 가능하게 (채택)

## 열린 질문

- Figma 입력을 읽는 방법 (Figma MCP 연결 여부)
- 프로젝트 경로 매핑의 구체 규칙 (Jira 라벨 네이밍 컨벤션 확정 필요)
- pr-reviewer 에이전트의 체크 기준 상세화 (보안/성능까지 볼지)
- 프로젝트별 테스트 실행 명령 파악 방식 (컨벤션 문서에 기록하는 것으로 시작)
