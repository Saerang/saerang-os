# 도메인 문서 템플릿 + RAG 작성 규칙

> 모든 도메인 문서의 **엄격한 공통 골격** + 검색 가능하게(RAG) 채우는 규칙.
> 목표: 어떤 프로젝트가 와도 **같은 모양으로 이쁘게** 나오고, 나중에 이 문서로 질문하면
> 답이 되게 한다. 완성 예시는 example.md 참조.

## 목차
- 1. 왜 엄격한가 (RAG)
- 2. RAG 작성 규칙 (어떻게 채우나)
- 3. 청크 분할 (디테일 ↔ 검색)
- 4. 공통 골격 (모든 문서 동일)
- 5. 렌즈별 변형

---

## 1. 왜 엄격한가 (RAG)

이 문서들은 결국 **질의응답 지식베이스**다. RAG는 문서 전체가 아니라 **청크(섹션)**를 꺼내
답을 만든다. 그래서 "sensible default"가 아니라 **strict** — 모든 문서가 같은 헤딩·같은 순서여야
ⓐ 질문→섹션 라우팅이 되고 ⓑ 스캔도 쉽다. 헤딩을 바꾸거나 섹션을 빼지 않는다(없으면 "해당 없음").

## 2. RAG 작성 규칙 (어떻게 채우나)

- **자기완결 청크** — 각 섹션은 그것만 꺼내도 말이 되게. "위에서 말한 그것" 같은 원거리 참조 ✗
- **근거 인라인** — 사실마다 `(근거: 파일:라인)`를 그 줄에 — 꺼낸 청크가 출처를 들고 가야 함
- **섹션 첫 문장 = 요약** — 임베딩·미리보기 품질을 위해 각 섹션을 한 문장 요약으로 연다
- **별칭(alias)** — 질문어 ≠ 코드어 문제 해소. 용어집과 frontmatter에 동의어를 적는다
  (예: `결제 실패 / 정산 반려 → PaymentFailed`). RAG 적중의 핵심
- **예상 질문 의식** — 각 도메인에서 자주 나올 질문("이 상태에서 X하면? 이 값 바꾸면 어디 깨지나?
  이 규칙 왜 있나?")에 답이 되게 서술. 함정·불변조건·정책을 명시적으로
- **SSOT 비중복** — 같은 사실을 두 문서에 다르게 적지 않는다. 프론트는 백엔드 SSOT를 **링크**(재정의 ✗)

## 3. 청크 분할 (디테일 ↔ 검색)

**디테일은 많을수록 좋다 — RAG는 필요한 청크만 꺼내므로 토큰 비용이 검색 시점에만 난다**
(progressive disclosure와 동형). 분량이 커지면 README 하나에 욱여넣지 말고 주제 파일로 쪼갠다:

```
docs/domain/<context>/
  README.md   ← 허브: frontmatter · 개요 · 서브도메인 · 경계 · 용어집(별칭) · 청크 인덱스(링크)
  model.md    ← 핵심 모델 · 애그리거트            ┐
  rules.md    ← 불변조건 · 비즈니스 규칙 · 정책     ├ 청크 (디테일)
  flows.md    ← 주요 흐름 · 도메인 이벤트          ┘
  (작으면 전부 README.md 한 파일로 둬도 됨)
```
파일/섹션 경계 = 청크 경계. README는 허브로서 각 주제 파일을 **직접 링크**(1단계).

## 4. 공통 골격 (모든 문서 동일)

```markdown
---
context: <영문 슬러그>
method: ddd | usecase | data-centric
role: backend | frontend | contract
subdomain: core | supporting | generic | [확인 필요]
stack: <예: spring-jpa>
owner_repo: <repo 이름>
base_commit: <hash>
updated: YYYY-MM-DD
aliases: [<동의어/업무어>, ...]
consumes: [<컨텍스트>(<관계패턴>), ...]   # 예: user(ACL)
---

# <도메인 한글명> (<영문 슬러그>) 도메인
> /domain 생성·사람 검수 | 기준 커밋 <hash>

## 개요               한 문단: 무엇이고 시스템에서 어떤 역할인가
## 서브도메인 분류      Core/Supporting/Generic + 근거 (Core면 가장 깊게)
## 바운디드 컨텍스트 경계  경계와 근거(언어/팀/변경속도/트랜잭션/데이터소유). [제안]은 열린질문으로
## 유비쿼터스 랭귀지     | 용어 | 코드 식별자 | 별칭 | 의미 |   ← 별칭 필수(RAG)
## 핵심 모델 · 애그리거트  루트·일관성 경계·상태전이·ID참조 여부 (근거)
## 불변조건            어기면 버그인 것 (근거)
## 비즈니스 규칙·정책    검증/계산/권한, 상수는 값+의미 (근거)
## 도메인 이벤트        의미 있는 사건(과거형)·발행/구독 — 있으면 (근거)
## 주요 흐름           유스케이스별 단계·코드 경로 (근거)
## 컨텍스트 맵 관계      다른 컨텍스트·프론트와의 이음새 패턴+방향 (근거)
## 외부 연동           대상·호출 지점·실패 동작 (근거)
## 함정·엣지 케이스      놓치기 쉬운 전제·테스트가 지키는 케이스 (근거)
## 미확인·열린 질문·경계 개선  [확인 필요] / [제안]
## 관련 코드 맵         주요 디렉터리/파일 + 한 줄 설명
```

## 5. 렌즈별 변형

골격은 같되, 선택된 method 렌즈에 따라 살을 다르게:
- **ddd**: 위 전체 (애그리거트·BC 경계·컨텍스트 맵 강조)
- **usecase**: "핵심 모델·애그리거트"→"주요 모델", "BC 경계"→간단히. **주요 흐름**을 가장 두껍게.
  애그리거트/컨텍스트맵 ceremony는 "해당 없음" 가능
- **data-centric**: "핵심 모델"→**스키마·제약·ER 관계**, 규칙은 **DB 제약 기반**으로. 애그리거트 생략 가능
공통 유지(렌즈 무관): frontmatter · 유비쿼터스 랭귀지(별칭) · 비즈니스 규칙 · 미확인·열린질문.
