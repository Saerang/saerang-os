# 추출 규율 (방법론 무관 — 항상 적용)

> domain-analyst가 *어떤 렌즈를 쓰든* 늘 지키는 규율. "무엇이 진짜 도메인 지식이고,
> 무엇은 노이즈/거짓인가"를 정한다. 모르는 프로젝트의 정찰(recon)도 여기 산다.

## 목차
- 0. 도메인 문서의 진짜 금 = 비즈니스 로직 (룰북)
- 1. 관련성 필터 — 무엇을 담고 무엇을 버리나
- 2. 우선순위 신호 — 어디를 깊게 파나
- 3. 죽은 코드 / 도달 가능성
- 4. 근거 규칙 (마커 + 확신도)
- 5. 정찰(recon) — 모르는 프로젝트 지도 그리기
- 6. 멀티소스 + surface 정량화 먼저 (breadth)
- 7. 비즈니스 로직을 룰북으로 번역 (depth)

## 0. 도메인 문서의 진짜 금 = 비즈니스 로직 (룰북)

이 문서가 존재하는 이유는 **AI/개발자가 도메인 규칙을 정확히 알고 구현**하게 하는 것이다.
구조도·클래스 목록·흐름도는 *오리엔테이션*일 뿐, 진짜 가치는 **결정 로직**이다:
"어떤 조건에서 / 무엇이 / 어떻게 계산·분기되는가". 그래서 모든 문서는 **breadth(넓게 훑기)와
depth(규칙은 깊게)를 동시에** 달성해야 한다 — 6절(멀티소스·정량화)로 넓이를, 7절(룰북 번역)로 깊이를.
코드 구조를 옮기면 "개발자 BC"가 되어 실패다.

---

## 1. 관련성 필터 — 무엇을 담고 무엇을 버리나

**핵심 질문**: *"이걸 모르면 버그를 낼까?"* → 그렇다면 도메인 지식.
*"코드 1분 읽으면 자명한가?"* → 노이즈. **문서는 코드를 다시 쓰는 게 아니라,
코드만 봐선 모르는 것(왜·전제·함정·정책)을 담는다.**

| 담을 것 (비자명한 업무 지식) | 버릴 것 (노이즈) |
|---|---|
| 불변조건 (어기면 버그) | getter/setter·DTO 매핑 |
| 정책·엣지케이스 (happy path 밖) | 프레임워크 배선·config |
| 상태머신 (불법 전이) | 평범한 CRUD 스캐폴딩 |
| 횡단 규칙 ("테스트 계정 제외" 등) | 로깅·직렬화 |
| 오해 부르는 용어 (코드어 ≠ 업무어) | 순수 인프라 어댑터\* |

\* 단, 인프라에 **업무 정책**이 박혀 있으면(재시도 3회 = 결제 PG 정책) 담는다.

## 2. 우선순위 신호 — 어디를 깊게 파나

코드에서 측정 가능한 "중요도" 신호. Core 서브도메인 + 아래 신호가 겹치는 곳을 가장 깊게:
- 🔥 **git churn** 잦음 = 뜨거운 도메인 (`git log --format= --name-only | sort | uniq -c | sort -rn`)
- 🧩 **복잡도 핫스팟** = if/예외/분기 몰린 곳에 규칙이 산다
- 🧪 **테스트 밀도** 높음 = 중요 규칙 (테스트는 규칙의 1급 근거)
- 🔗 **의존 중심성** 높음 = 핵심 모델
- 🎯 **Core 서브도메인** = method-ddd 1.3 기준 가장 깊게, Generic은 얇게

## 3. 죽은 코드 / 도달 가능성

죽은 코드의 규칙을 본문 사실로 적으면 **거짓 도메인 지식**이 된다. 무시하면 누락.

**탐지 단서**: 호출처 없음(grep 참조 0) · `@Deprecated` · 영구 false 플래그 뒤 ·
미참조 엔드포인트 · 테스트도 호출처도 없음 · git에서 수년째 미변경.

**⚠️ 가짜 죽음 함정** (정적으론 죽어 보이나 살아있음): DI·리플렉션·프레임워크 진입점 ·
외부 공개 API · 이벤트 핸들러 · 동적 디스패치. → **"호출처 없음" ≠ 확실히 죽음.**

**처리 규칙**: 본문 사실로 적지 말고 **`[확인 필요]` "도달 가능성 불확실(호출처 없음) —
살아있는 규칙인지 확인"** 으로. 절대 단정하지도, 조용히 버리지도 않는다.

## 4. 근거 규칙 (마커)

- 본문의 모든 규칙·불변조건·정책·흐름은 **`(근거: 파일:라인)`** 필수. 없으면 본문 ✗, "열린 질문"으로
- **`[추정]`** — 코드에서 유추한 의도·배경 (근거는 대되 단정 ✗)
- **`[제안]`** — DDD/설계 처방 (경계를 다르게 긋자 등). "열린 질문 · 경계 개선"에만. 사람이 채택해야 승격
- **`[확인 필요]`** — 코드로 확인 못 했거나 모순. 도메인 전문가에게 물을 것
- **확신도 N(근거)** — 한 줄 결론·요약·핵심 규칙 문장엔 확신도(0~100)+근거를 단다(C8DEV식).
  직접 코드 확인=85~95, 강한 유추=60~80, 이름만 보고=30 이하(스텁으로 표기)
- 일반론·도메인 상식("보통 전자상거래는~")을 이 프로젝트 사실로 적지 않는다

## 5. 정찰(recon) — 모르는 프로젝트 지도 그리기

낯선 코드베이스는 **가정 금지, 지도부터.** 깊은 추출 전에 싸게 훑어 *제안*만 만든다:

| 정찰 항목 | 방법 |
|---|---|
| **스택 판별** | 빌드파일(pom/gradle/package.json)·의존성·확장자 |
| **영토 매핑** | 최상위 모듈/패키지 → 바운디드 컨텍스트·기능 후보 |
| **도메인 vs 인프라** | 헥사고날 로케이터 — 중심(도메인 모델·서비스) / 가장자리(컨트롤러·어댑터·repo) |
| **방법론 적합도** | 규칙 풍부·애그리거트·상태머신 → DDD / 빈약+CRUD → data-centric / 기능핸들러 중심 → use-case |
| **핫스팟 우선순위** | 2절 신호로 어디부터 깊게 |
| **진입점·저장소** | 컨트롤러/라우트, DB 스키마/마이그레이션 |

산출: **도메인 맵 초안** — 컨텍스트/기능 후보 + 제안 렌즈 + 스택 + 역할 + 우선순위.
경계가 불확실하면 후보로만 제시하고 사람 확정을 받게 한다(지어내지 않는다).

## 6. 멀티소스 + surface 정량화 먼저 (breadth)

**단일 모듈만 읽으면 breadth를 놓친다.** "엄청 많은 것"의 실체(전 타입·전 캠페인·전 설정)는
서비스 코드가 아니라 표면·데이터·운영에 산다. 깊이 들어가기 전에 **넓이를 먼저 센다:**

| 소스 | 무엇을 얻나 |
|---|---|
| **API 표면** (전 채널 컨트롤러·엔드포인트, 이벤트/캠페인 모듈) | 이 도메인이 노출하는 기능의 전체 수·종류 |
| **DB 스키마/DDL/데이터** | 실제 데이터 모델·코드/구분값의 실제 가짓수 (breadth의 1급 근거) |
| **DB 저장 프로시저·트리거·함수** | ⚠️ **레거시 비즈니스 로직의 대형 은닉처.** 월배치 지급·캠페인·정산·증정 데이터 적재가 애플리케이션 코드가 아니라 SQL 프로시저/트리거에 사는 경우가 흔하다. **앱 코드(Java 등)만 보면 도메인의 절반을 놓친다.** "procedure-migration" 류 브랜치·`USP_*`/`ufn_*`·`TR_*`가 있으면 반드시 읽는다 |
| **시드/마이그레이션 데이터** | test-data.sql·시드 INSERT는 실제 인스턴스(쿠폰 행 등)의 *구체 변종*을 보여줌 (breadth) |
| **admin/운영 모듈** | 운영자가 만드는 "많은 것"(쿠폰·캠페인·설정)의 구성 화면·규칙 |
| **캠페인/이벤트 클래스** | 개별 프로모션(예: PoaXXXX)들 — 제너릭 엔진이 아니라 *구체 사례*가 breadth |
| **서비스 코드** | 메커니즘(엔진) |
| **enum/상수/config** | 분기의 가짓수와 임계값 |

먼저 **정량화**한다: "이 도메인 = 엔드포인트 N개 · 테이블 M개 · 타입 K종 · 캠페인 P개".
그리고 그 많은 것들을 **카탈로그로 열거**한다(엔진만 서술하고 끝내지 말 것).

## 7. 비즈니스 로직을 룰북으로 번역 (depth)

코드 인용("BonusPackCalculator.java:89")만으론 규칙을 모른다. **업무 규칙으로 번역**한다:

- **조건 → 결과** 문장: "구매 수량이 (구매+증정)의 배수이고 한도 이내면 → 증정 위치 제품을 무료 처리"
- **결정표**: 타입/조건별로 무엇이 어떻게 계산·분기되는지 표로 (정액/정률/보너스팩 각각)
- **구체 예시**: "3+1 쿠폰에 4개 구매 → 4번째 무료 / 5개 구매 → 혜택 0 (배수 아님)"
- **임계값·상수의 의미**: 매직넘버는 값+업무 의미("재시도 3회 = PG 정책")
- **타입/캠페인별로** 자격(누가)·조건(언제)·계산(어떻게)·한도·엣지를 각각 — 제너릭 1벌로 뭉치지 말 것
- 코드 근거는 유지하되(`(근거: 파일:라인)`), **문장은 업무 언어로**. 확신 없으면 7절도 [추정]/[확인 필요]
