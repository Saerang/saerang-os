## 0. 서술 vs 처방 (마커 규칙 — 가장 중요)

이 시스템의 원칙은 "코드에서 확인된 사실만". DDD엔 설계 제안(처방)이 섞이므로 둘을 분리한다:

- **서술(사실)** — 코드에서 *복원한* 것. `(근거: 파일:라인)` 필수. 본문에 그대로.
- **`[추정]`** — 코드에서 유추한 의도·배경. 근거는 대되 단정하지 않음.
- **`[제안]`** — "이렇게 가르는/묶는 게 낫다"는 **설계 처방**. 코드 사실이 아니라
  DDD 관점의 개선안. 반드시 이 마커를 붙이고 "미확인·열린 질문 / 경계 개선 제안"에만 적는다.
  사람이 채택해야 사실로 승격된다.
- **`[확인 필요]`** — 코드로 확인 못 했거나 모순. 도메인 전문가에게 물을 것.

규칙: **컨텍스트 경계·애그리거트 경계를 코드와 다르게 그리는 모든 판단은 `[제안]`이다.**

## 1. 전략적 설계

### 1.1 바운디드 컨텍스트 — 경계 찾는 5신호

바운디드 컨텍스트 = **하나의 모델이 일관되게 통하는 언어적 경계**. 같은 용어가 경계 밖에선
다른 뜻이 된다. **코드의 패키지 경계 ≠ 컨텍스트 경계** — 아래 신호로 진짜 경계를 추론한다:

| 신호 | 코드에서 탐지하는 법 |
|---|---|
| 🗣️ **언어 경계**(최강) | 같은 단어가 다른 모델로 두 번 정의됨 (예: `Order`가 sales와 shipping에 각각). 같은 클래스명·다른 필드셋, 또는 같은 개념·다른 이름 |
| 👥 **팀/소유 경계** | 디렉터리 소유자(CODEOWNERS), 별도 배포 단위, 모듈 간 의존 방향 |
| 🔄 **변경 속도** | git 히스토리상 함께 바뀌는 파일끼리 묶이고, 따로 바뀌면 갈린다 |
| 🔒 **트랜잭션 경계** | 한 트랜잭션/한 일관성 단위로 묶이는 범위 (`@Transactional` 범위, 한 커밋으로 저장되는 모델) |
| 📊 **데이터 소유권** | 이 개념의 SSOT가 누구인가 — 누가 쓰고 누가 읽기만 하나 (FK 방향, API 호출 방향) |

코드 경계와 도메인 경계가 어긋나면 → 사실은 그대로 기록하고, 더 나은 분할은 **`[제안]`**.

### 1.2 유비쿼터스 랭귀지 — 용어 충돌 = 경계 신호

개발자·도메인 전문가가 공유하는 **하나의 엄밀한 언어**. 코드 식별자가 이 언어여야 한다.
- 용어집은 **1급 산출물**: `용어 | 코드 식별자 | 이 컨텍스트에서의 의미`.
- **핵심 활용**: 같은 용어가 컨텍스트마다 다른 의미면 → **그 자체가 경계 증거**(1.1의 언어 경계).
  크로스 컨텍스트로 용어를 모아 충돌을 찾으면 경계가 드러난다.
- 코드 식별자와 업무 용어가 어긋나면(예: `tmp_user`가 실은 "초대 대기자") → 불일치를 기록.

### 1.3 서브도메인 분류 — Core / Supporting / Generic

모든 도메인을 같은 깊이로 다루지 말고, **Core에 깊이를 몰아준다.**

| 분류 | 정의 | 코드 단서 | 문서 깊이 |
|---|---|---|---|
| 🎯 **Core** | 경쟁우위·차별점. 이 사업의 이유 | 가장 복잡한 규칙·자주 바뀌는 핵심 로직·커스텀 알고리즘 | 가장 깊게 |
| 🔧 **Supporting** | 필요하나 차별점 아님 | 사업 특화지만 평범한 CRUD/워크플로 | 보통 |
| 📦 **Generic** | 어디나 있는 것 (인증·이메일·결제연동·파일저장) | 외부 라이브러리·표준 패턴·서드파티 SDK 래핑 | 얇게 (사 쓰는 영역) |

휴리스틱(Evans): Core는 대략 전체 가치의 ~20%, 코드의 ~5%, 노력의 ~80%를 차지한다 —
즉 **작지만 가장 중요**. 다른 영역 투자는 Core를 어떻게 받치는지로 정당화한다.

## 2. 전술적 설계

### 2.1 애그리거트 — Vernon 4규칙 + 코드 탐지

애그리거트 = **하나의 트랜잭션으로 일관성을 지키는 경계**(루트 엔티티로만 접근). Vernon 4규칙:

1. **진짜 불변조건만 일관성 경계 안에** — 한 트랜잭션에서 반드시 함께 맞아야 하는 것만 한 애그리거트로.
2. **애그리거트는 작게** — 큰 애그리거트는 성능·동시성 안티패턴.
3. **다른 애그리거트는 ID로 참조** — 객체 그래프로 물지 말고 식별자로. (코드 단서: 직접 객체 참조 vs `xxxId` 필드)
4. **다른 애그리거트는 결과적 일관성으로 갱신** — 도메인 이벤트로 비동기 갱신.

코드 탐지: `@Transactional` 범위, 리포지토리 단위(보통 1 애그리거트 = 1 리포지토리),
캐스케이드 저장 범위, 락 범위가 애그리거트 경계의 단서. 경계가 부적절하면 → **`[제안]`**.

### 2.2 엔티티 vs 값 객체
- **엔티티**: 식별자로 구분, 생애주기 동안 변함 (ID 있음).
- **값 객체**: 속성으로 정의, 불변, 교체로 변경 (예: Money, Address). 코드 단서: equals/hashCode가
  모든 필드 기반·불변(final)·ID 없음.

### 2.3 도메인 이벤트
"도메인에서 일어난 의미 있는 사건"(과거형: `OrderPlaced`, `PaymentFailed`). 코드 단서:
이벤트 발행/구독, 메시지 큐, `*Event` 클래스. 애그리거트 간 결과적 일관성의 매개체이자
컨텍스트 간 통합(Published Language)의 단위.

## 3. 컨텍스트 매핑 — 9패턴 카탈로그 + 코드 단서

컨텍스트 간 **관계의 종류**. back↔front, back↔back 이음새가 어떤 패턴인지가 멀티프로젝트의 핵심.
**상류(Upstream, U)**는 영향을 주고 **하류(Downstream, D)**는 받는다(하류는 상류에 영향 못 줌).

| 패턴 | 방향 | 정의 | 코드 단서 |
|---|---|---|---|
| **Partnership** | 대칭 | 두 컨텍스트가 함께 성패 — 조율해서 같이 변경 | 양방향 의존·동시 릴리스 |
| **Shared Kernel** | 대칭 | 모델/코드 일부를 **공유**(위험, 명시적 합의 필요) | 공유 모듈/라이브러리에 도메인 모델이 들어 있음 |
| **Customer-Supplier** | U→D | 하류 요구가 상류 계획에 **반영됨**(상류가 들어줌) | 하류 요구로 상류 API가 진화한 흔적 |
| **Conformist** | D | 하류가 상류 모델을 **번역 없이 그대로** 따름(협상력 없음) | 상류 DTO/스키마를 그대로 도메인에 사용 |
| **Anticorruption Layer (ACL)** | D | 하류가 **번역층**을 둬 상류 모델 오염 차단 | adapter/translator/mapper/`*Client` 래퍼 계층 |
| **Open Host Service (OHS)** | U | 상류가 **공개 프로토콜**로 다수 하류에 서비스 | 공개 REST/gRPC API, 버전드 엔드포인트 |
| **Published Language (PL)** | U | 공용 **계약 언어**(OpenAPI·GraphQL 스키마·이벤트 스키마) | 스펙 파일, 공유 스키마 레지스트리 |
| **Separate Ways** | — | 통합 안 함, 독립 | 연결 없음 |
| **Big Ball of Mud** | — | 경계 없는 진흙공(인식용) | 모델이 뒤엉켜 경계 식별 불가 → `[확인 필요]` |

상류 역할 = OHS·PL / 하류 역할 = Conformist·ACL.

**프론트엔드 적용**: 프론트는 거의 항상 백엔드의 **하류**다. 프론트가 백엔드 응답을 그대로 쓰면
**Conformist**(백엔드 변경에 취약), 번역층(api-client + view-model 매핑)을 두면 **ACL**.
back↔front 이음새가 OpenAPI 등 스펙이면 **Published Language**. 프론트 도메인 문서엔
"order 컨텍스트를 Conformist/ACL 중 무엇으로 소비하는가"를 적어야 변경 파급이 보인다.

## 4. 분석 적용 순서

1. **언어부터** — 용어집을 만들며 같은 단어의 다중 의미를 찾는다(경계 신호).
2. **경계 긋기** — 1.1의 5신호로 바운디드 컨텍스트를 식별. 코드와 어긋나면 사실+`[제안]`.
3. **서브도메인 분류** — Core를 찾아 깊이를 배분.
4. **내부 전술** — 애그리거트·불변조건·엔티티/값객체·도메인 이벤트를 코드 근거와 함께.
5. **외부 관계** — 컨텍스트 맵 패턴으로 다른 컨텍스트·프론트와의 이음새를 분류.
6. 확신 못 하는 경계·관계는 `[확인 필요]`로 남긴다 — 지어내지 않는다.
