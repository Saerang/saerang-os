# Domain (도메인 지식 문서 구축·갱신)

"도메인 지식이 없으면 AI가 코딩 못 한다"를 해결하기 위해, 대상 프로젝트의 코드에서
도메인 지식을 추출해 `docs/domain/<도메인>/`으로 버전관리하는 스킬.
**코드에서 확인된 사실만 문서가 된다** — 추측은 사람 검수를 통과해야만 사실로 승격된다.

문서 구조: 도메인당 디렉토리 하나 — 본 문서는 `docs/domain/<도메인>/README.md`,
분량이 커지면 같은 디렉토리에 주제별 파일(model.md, rules.md, flows.md 등)로 분할한다.
바운디드 컨텍스트를 소유한 repo의 문서가 원본(SSOT)이다.
모든 문서는 `references/doc-template.md`의 **엄격한 공통 골격 + RAG 작성 규칙**을 따른다 —
나중에 이 문서로 질의응답(RAG)하기 위해 일관된 형식·자기완결 청크·별칭이 필수다.

**목표 수준 = 코드레벨 구현 지식 (coding-ready).** 이 스킬의 산출물은 비즈니스 개요가 아니라
**개발자/AI가 소스를 안 열고 이 문서만으로 해당 도메인에 기능을 추가·수정할 수 있는 코드레벨 레퍼런스**다:
엔티티 필드·타입·DB컬럼, enum 값 전수, 규칙 공식·임계값·예시, 상태 전이, 흐름(진입점→클래스·메서드),
예외 타입·조건, SQL 레이어(프로시저·트리거)까지 — 전부 `(근거: 파일:라인)`. 판정 기준은 doc-template 4.5
"코딩 레디 루브릭". 구조는 `method-*.md` 렌즈(기본 DDD)로 잡되 **깊이는 코드레벨이 기본값**이다.
참조: extraction-discipline(관련성·멀티소스·SQL은닉·규율) / doc-template(골격·RAG·루브릭) /
method-ddd(렌즈) / example(본보기).

## 인자 / 모드 판별

`$ARGUMENTS`(스킬 인자)를 확인한다:

- 토큰 중 **실재하는 디렉터리 경로**가 있으면 → 대상 프로젝트 경로. 없으면 사용자에게
  물어본다 (이 OS 저장소 자신은 대상이 아니다 — 도메인 문서는 대상 프로젝트 repo에 산다)
- 나머지 토큰이 **도메인 이름**이다. 여러 개면 각각 분석한다 (분석은 병렬, 검수는 순차)
- 도메인 이름이 없거나 도메인 구획 자체를 모르면 → **recon 모드**: domain-analyst를 recon으로
  돌려 도메인 맵 초안(컨텍스트 후보·제안 렌즈·스택·우선순위)을 받아 사용자에게 보여주고,
  어떤 도메인부터 다룰지·방법론 렌즈를 확정받는다 (모르는 프로젝트의 진입점).
  기존 문서가 있으면 `docs/domain/` 목록도 함께 보여준다
- **모드는 기존 문서 존재 여부로 자동 판단한다** (README.md만 보지 말 것 — `ls docs/domain/<도메인>/`로
  `*.md` 전부 확인):
  - 기존 문서가 **하나도 없음** → **bootstrap 모드** (신규 구축)
  - 기존 문서가 **하나라도 있으면**(README.md·**domain.md**·기타 *.md) → **update 모드**: 그 문서(들)를
    **베이스로 읽어** 갱신·정합화한다. **팀/타 도구가 만든 기존 문서도 반드시 베이스로 삼는다** —
    무시하고 새로 쓰면 중복·충돌·후퇴가 난다 (실패 사례: 기존 domain.md를 못 보고 더 얇은 문서를 새로 씀)
  - 인자가 `rebuild`로 시작하면 → 문서가 있어도 **bootstrap 전면 재구축**
    (기존 문서는 참고 자료로 전달). 모드 판정 결과를 사용자에게 한 줄로 알린다

## 절차

1. **도메인 분석 (domain-analyst 서브 에이전트)** — **domain-analyst 서브 에이전트**
   (Agent 도구, subagent_type: `domain-analyst`)에 다음을 전달해 분석을 맡긴다.
   도메인이 여러 개면 병렬로 spawn한다:
   - 모드 (recon | bootstrap | update), 대상 프로젝트 경로, 도메인 이름
   - **레퍼런스 경로** (`.claude/skills/domain/references/` 아래): `extraction-discipline.md`(항상)
     + `doc-template.md`(bootstrap/update) + `method-<렌즈>.md`(확정 렌즈, 기본 `method-ddd.md`)
     + `example.md`(형식 본보기)
   - **방법론 렌즈** ddd|usecase|data-centric — recon 결과로 제안받아 사람이 확정한다.
     쓰려는 렌즈의 `method-*.md`가 아직 없으면 만들지 말고 `method-ddd.md`로 진행하거나 사용자에게 알린다
   - 역할 backend|frontend|contract — 프론트는 백엔드 SSOT를 링크 참조만 (재정의 ✗)
   - update 모드면: **기존 문서 전체(`docs/domain/<도메인>/*.md` — domain.md 등 팀 작성분 포함)를
     베이스로 전달**하고, 문서 헤더의 **기준 커밋 해시**(없으면 `git log -1 --format=%H -- docs/domain/<도메인>/`).
     분석가는 기존 내용을 재검증·보완하되 근거 없이 삭제하지 않는다
   - "분석 불충분"이 반환되면 사유를 보고하고 종료한다 (도메인이 크면 분할 재요청 안내)

2. **초안 자체 점검** — 에이전트가 반환한 초안(bootstrap) 또는 섹션별 변경
   제안(update)을 받아 점검한다:
   - 본문 항목에 근거 표기 `(근거: 파일:라인)`가 빠진 곳이 있으면 해당 항목을
     "미확인·열린 질문"으로 강등시킨다 (에이전트가 규칙을 어겼어도 여기서 한 번 더 거른다)
   - **`[제안]`(DDD 설계 처방)이 본문 사실에 섞여 있으면** "경계 개선 제안"으로 옮긴다 —
     설계 처방이 코드 사실로 위장되지 않게 한다
   - 빈 섹션을 일반론으로 채운 흔적이 있으면 삭제한다
   - **코딩 레디 점검** (이 스킬의 합격선): doc-template 4.5 루브릭 — 엔티티 필드·enum 값 전수·
     규칙 공식/임계값·상태 전이·흐름(진입점→메서드)·예외 타입·SQL 레이어·근거 file:line이 갖춰졌나.
     미달 항목은 [확인 필요]로 명시(빈칸 금지). 마크다운 표는 헤더·행 셀 수 일치(렌더 깨짐 방지)

3. **사람 검수 게이트 (저장 전 필수 — 생략 금지)** — 도메인별로 순차 진행한다:
   - 초안 전문(update면 변경 제안 전문)을 사용자에게 보여준다
   - **`[추정]`/`[확인 필요]`/`[제안]` 항목을 AskUserQuestion으로 전수 처리한다**
     (한 번에 4개 이하로 묶어서):
     - `[추정]`/`[확인 필요]`: "사실 맞음(확정) / 수정 필요(직접 입력) / 틀림(삭제) /
       나도 모름([확인 필요]로 문서에 남김)" 중 선택
     - `[제안]`(DDD 경계·애그리거트 처방): "채택(사실로 승격) / 보류([제안]으로 남김) /
       버림" 중 선택
   - update 모드의 **[삭제 제안]**(코드에서 사라진 규칙)은 반드시 개별 확인을 받는다 —
     문서에서 지식을 지우는 것은 추가보다 위험하다
   - 추측 항목 처리가 끝나면 AskUserQuestion으로 **"이대로 저장할까요?"** 최종
     확인을 받는다. 수정 의견이 있으면 반영 후 다시 확인받는다

4. **저장** — 검수 통과본을 대상 프로젝트의 `docs/domain/<도메인-영문-슬러그>/README.md`로
   저장한다 (디렉토리 없으면 생성). doc-template의 frontmatter(context·method·role·subdomain·
   stack·aliases·base_commit·consumes)를 채우고, 분량이 큰 섹션은 주제별 파일(model/rules/flows.md)로
   분할해 README.md에서 링크한다. 기준 커밋은 현재 HEAD(`git rev-parse --short HEAD`), 갱신일은 오늘.

5. **커밋 제안** — AskUserQuestion으로 커밋 여부를 물어본다:
   - **예** → 대상 프로젝트에서 `docs/domain-<도메인>` 브랜치를 만들어 커밋한다
     (메시지는 commit 스킬 스타일의 한글, 예: `주문 도메인 문서 추가`).
     커밋 후 push·PR 생성 여부를 이어서 물어본다 — 도메인 문서 PR은 팀 동료가
     내용을 한 번 더 검수하는 2차 게이트 역할을 한다
   - **아니오** → 파일 저장까지만 하고 종료

6. **완료 보고** — 문서 경로, 모드, 검수에서 확정/수정/삭제된 추측 개수,
   `[확인 필요]`로 남은 항목 수, 커밋/브랜치 상태를 한글로 보고한다.
   `[확인 필요]`가 남아 있으면 "도메인 전문가에게 물어볼 것" 목록으로 정리해 준다.

## 주의사항

- **검수 없이 저장하지 않는다** — 어떤 경우에도 3단계(사람 검수 게이트)를 건너뛰고
  파일을 쓰지 않는다. 이 문서는 ticket-analyst와 develop이 사실로 믿고 소비하는
  SSOT이므로, 검증되지 않은 추측이 저장되는 것이 이 스킬의 최대 실패 모드다
- 대상 프로젝트의 main/master에 직접 커밋·push하지 않는다 — 반드시 브랜치에서 작업
- **도메인 문서 외의 파일은 건드리지 않는다** — 분석 중 발견한 코드 문제는
  보고만 하고 수정하지 않는다
- update 모드에서 기존 문서의 사람 작성 내용(특히 검수에서 확정됐던 항목)은
  코드 근거 없이 삭제하지 않는다 — 삭제는 항상 개별 확인 후에만
- 사용자도 모르는 항목은 지어내지 말고 `[확인 필요]`로 남긴다 — 구멍이 보이는
  문서가 매끈한 거짓 문서보다 낫다
- 도메인 이름은 팀이 부르는 이름(한글 가능)을 그대로 쓰되, 디렉토리명은 영문 슬러그를
  권장한다 (예: `주문` → `docs/domain/order/README.md`, 문서 제목에 한글 병기)
- Jira/Confluence가 연동되면 코드 외 소스(기획 문서, 과거 티켓)도 분석 입력으로
  확장한다 — 현재는 코드 + 대상 repo 내 로컬 문서만 소스로 쓴다
- 보고와 문서는 한글로 작성한다 (코드 식별자, 고유명사는 원문 유지)
