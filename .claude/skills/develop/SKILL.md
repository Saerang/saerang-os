# Develop (티켓 → PR 개발 오케스트레이터)

티켓 하나 = PR 하나를 원칙으로, 구현부터 PR 생성까지 책임지는 오케스트레이터.
기획 분해(티켓 생성)와 PR 머지 판단은 사람의 몫이며, 이 스킬은 그 사이를 맡는다.

## 인자 처리

`$ARGUMENTS`(스킬 인자)를 확인한다:

- 인자가 `check`로 시작하면 → **체크 모드**: 뒤에 오는 티켓들(여러 개 가능)을
  ticket-analyst로 **병렬 분석만** 하고, 티켓별 "진행 가능 / 보강 필요 + 질문"을
  보고한 뒤 종료한다 (구현 없음 — 사전 스크리닝용)
- 첫 토큰이 Jira 티켓 키(예: `ABC-123`)면 → Jira에서 티켓을 읽는다
  (Jira 도구 호출이 실패하거나 도구가 없으면 그 사실을 알리고 기획 내용을 직접 달라고 요청한다)
- 파일 경로면 → 해당 기획 문서를 읽는다
- 둘 다 아니면 → 인자 전체를 **기획 내용 자체**로 취급한다 (Jira 없이 직접 전달)
- 두 번째 인자로 프로젝트 경로가 오면 그 경로를 대상 프로젝트로 사용
- 인자가 없으면 어떤 티켓/기획을 개발할지 물어본다

## 절차

1. **티켓 분석 (ticket-analyst 서브 에이전트)** — 대상 프로젝트 경로를 먼저 결정한 뒤
   (① 인자로 받은 경로 → ② 티켓 라벨/필드 → ③ 둘 다 없으면 사용자에게 질문),
   **ticket-analyst 서브 에이전트**(Agent 도구, subagent_type: `ticket-analyst`)에
   티켓 원문과 프로젝트 경로를 전달해 분석을 맡긴다:
   - **보강 필요** → 질문 목록을 사용자에게 보고하고 **오케스트레이터를 종료한다**
     (fail-fast — 티켓 품질 게이트. 사람이 티켓을 보강한 뒤 다시 실행하면 됨)
   - **진행 가능** → 스펙 요약을 보여주고 AskUserQuestion으로 **"이대로 진행?"을
     확인받는다 (시작 승인 게이트)** — 승인되면 이후 단계는 무인으로 끝까지 진행,
     수정 의견이 있으면 반영 후 진행. 이미 check 모드로 승인된 티켓이면 이 게이트는 생략

2. **컨텍스트 로딩** — ticket-analyst가 보고한 컨벤션 문서와 도메인 문서를 읽는다:
   - **도메인 문서가 보고되면** 해당 README(및 프론트매터 `consumes`에 적힌 연관 컨텍스트
     문서)를 읽어 불변조건·비즈니스 규칙·용어를 구현 컨텍스트로 로딩한다 — 이 SSOT의
     규칙을 위반하지 않게 구현한다. 신선도가 "오래됨"으로 보고됐으면 사용자에게
     `/domain update` 선행을 권하되 진행을 막지는 않는다 (문서가 코드보다 뒤처졌을 수 있음)
   - 컨벤션 문서가 **없다고 보고되면** 기존 코드 패턴(구조, 네이밍, 테스트 방식)을 분석해
     디폴트 컨벤션 md를 생성·추가하고 사용자에게 알린다
   - 테스트 실행 방법이 미확인이면 직접 파악한다 (package.json, Makefile, CI 설정 등)

3. **계획 수립** — PR 하나 기준으로 작업 계획을 세운다:
   - 브랜치 생성: 티켓 키 포함 (예: `feat/ABC-123-로그인-개선`).
     티켓 키가 없는 직접 기획이면 기획 요지로 브랜치명을 짓는다 (예: `feat/할인-쿠폰-적용`)
   - 작업량이 크고 독립적인 부분이 있으면 git worktree 병렬 진행을 판단하되,
     판단 근거를 사용자에게 한 줄로 알린다 (기본은 단일 진행)

4. **구현** — 컨벤션을 준수하며 구현한다. 커밋 메시지는 commit 스킬 스타일(한글)을 따른다.

5. **검증** — PR 전 품질 게이트 (둘 다 필수):
   - 테스트 실행 → 실패 시 수정 후 재실행
   - **pr-reviewer 서브 에이전트**(Agent 도구, subagent_type: `pr-reviewer`)에 변경 diff와
     컨벤션 문서 경로, **그리고 step 2에서 로딩한 도메인 문서 경로**(있으면)를 전달해 리뷰
     → 지적사항 반영 → 테스트 재확인. 도메인 문서를 넘기면 리뷰어가 스타일뿐 아니라
     도메인 불변조건·비즈니스 규칙 위반까지 검증한다

6. **PR 생성** — `gh` CLI로 PR을 올린다 (`gh`가 설치되어 있지 않으면 브랜치 push까지 하고,
   GitHub이 출력하는 PR 생성 URL과 준비된 PR 본문을 사용자에게 전달한다):
   - PR 본문(한글): 요약 / 변경사항 / 테스트 결과 / Jira 티켓 링크
   - **막혔을 때** (테스트 반복 실패, 의존성 부재, 기획 불명확으로 진행 불가 등):
     중도 포기하지 말고 그 시점까지를 **Draft PR**로 올린 뒤,
     시도한 것 · 막힌 이유 · 선택지를 정리해 보고한다

7. **도메인 문서 동기화 (피드백 루프)** — 이번 PR이 step 2에서 로딩한 도메인의
   불변조건·비즈니스 규칙·상태 전이·용어·주요 플로우를 **바꿨는지** 판단한다
   (단순 버그 수정·리팩터링·내부 구현 변경은 해당 없음 — 문서에 적힌 규칙 자체가
   변했을 때만):
   - **바뀌었으면** 어떤 도메인 문서의 어느 항목이 낡았는지 사용자에게 보고하고
     `/domain update <도메인>` 선행을 제안한다 (develop이 소비만 하고 끝나면 문서가
     PR마다 뒤처지므로, 생산자 역할로 루프를 닫는다). PR 본문에도
     "도메인 영향: <도메인> 규칙 변경 — 문서 갱신 필요" 한 줄을 남긴다
   - **안 바뀌었으면** 이 단계는 건너뛴다
   - 문서를 이 자리에서 직접 고치지는 않는다 — 도메인 문서는 `domain` 스킬의
     사람 검수 게이트를 거쳐야 저장되므로, 갱신은 그 스킬에 위임한다

8. **완료 보고** — PR URL, 검증 결과(테스트/리뷰 통과 여부),
   도메인 문서 영향(갱신 필요 여부), 남은 일(있다면)을 한글로 보고한다.

## 주의사항

- **티켓 하나 = PR 하나** — 범위가 넘치면 진행 전에 분할을 제안한다
- **기획을 대신 결정하지 않는다** — 분석 결과가 "보강 필요"면 질문 목록만 남기고 종료한다.
  단, 사소한 구현 디테일은 컨벤션과 기존 코드 패턴을 따라 스스로 결정한다
- 대상 프로젝트의 main/master에 직접 커밋·push하지 않는다 — 반드시 브랜치에서 작업
- **막혀도 중도 포기 금지** — Draft PR + 상황 보고가 실패 시의 정상 출구다
- 테스트를 통과시키기 위해 테스트를 약화시키거나 skip 처리하지 않는다 (치팅 금지)
- force push는 사용자가 명시적으로 요청할 때만
- 컨벤션 문서를 새로 생성했으면 PR에 포함하지 말지 사용자에게 확인한다
- 보고와 PR 본문은 한글로 작성한다 (코드 식별자, 고유명사는 원문 유지)
