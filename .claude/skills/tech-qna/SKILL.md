# Tech Q&A (검증된 기술 답변)

기술 질문에 대해 **실제 출처를 검증한 뒤** 답변하는 스킬. 기억에만 의존해 답하지 않는다.

## 모드 판별

`$ARGUMENTS`(스킬 인자)를 확인한다:

- 인자가 `detail`로 시작하면 → **상세 모드** (detail 뒤의 텍스트가 질문)
- 그 외 → **요약 모드** (인자 전체가 질문)
- 인자가 없으면 사용자에게 무엇이 궁금한지 물어본다

## 절차

1. **질문 분석** — 핵심 키워드와 검증이 필요한 사실(버전, 날짜, 스펙, 동작 방식 등)을 파악한다.

2. **검색 및 검증** (필수 — 생략 금지):
   - WebSearch로 질문 관련 자료를 검색한다 (필요하면 ToolSearch로 WebSearch/WebFetch를 먼저 로드)
   - 핵심 주장마다 **최소 2개 이상의 독립적인 출처**로 교차 확인한다
   - 공식 문서 > 공식 블로그/릴리즈 노트 > 신뢰할 수 있는 기술 블로그 순으로 신뢰한다
   - **시간에 민감한 사실(출시 상태 preview/GA, 버전, 가격, 스펙 한도)은 반드시 벤더의 공식 문서/릴리즈 노트를 WebFetch로 직접 조회해서 확인한다** — 서드파티 블로그/리뷰는 작성 시점의 스냅샷이라 몇 주만 지나도 틀릴 수 있다. 공식 출처를 못 찾았으면 "공식 출처 미확인"이라고 명시한다
   - 출처의 작성 시점을 확인한다 — 빠르게 바뀌는 주제라면 오래된 출처는 근거에서 제외하거나 시점을 함께 표기한다
   - 출처 간 내용이 다르면 **공식 문서를 우선**하고, 어느 쪽도 공식이 아니면 "검증 불가" 또는 "출처 간 상이"로 솔직하게 표시한다
   - 버전/날짜에 민감한 내용은 최신 정보인지 확인한다 (오늘 날짜 기준)

3. **답변 작성** — 모드에 따라:

### 요약 모드 (기본)

```
**요약**: 핵심 답을 1~2문장으로

**내용**:
- 알아야 할 핵심 포인트를 bullet 3~6개 정도로 적당히
- 너무 깊게 들어가지 않는다

**검증**: 교차 확인 결과 한 줄 (예: "공식 문서와 릴리즈 노트에서 확인됨")

**출처**:
- [제목](URL)
- [제목](URL)
```

마지막에 "더 자세한 내용이 필요하면 `/tech-qna detail <질문>`으로 물어보세요"라고 안내한다.

### 상세 모드 (`detail`)

```
**요약**: 핵심 답 1~2문장

**상세 내용**:
배경, 동작 원리, 코드 예시, 옵션 비교, 주의사항, 엣지 케이스까지
섹션을 나눠 깊이 있게 설명

**검증**: 각 핵심 주장별로 어떤 출처에서 확인했는지

**출처**:
- [제목](URL) — 무엇을 확인한 출처인지 한 줄 설명
- [제목](URL) — ...
```

## 주의사항

- **출처 없는 답변 금지** — 검색이 실패하면 그 사실을 알리고, 기억 기반 답변임을 명시한다
- 출처 URL은 실제로 검색/조회한 페이지만 적는다 (URL을 지어내지 않는다)
- 답변은 한글로 작성한다 (기술 용어, 코드, 고유명사는 원문 유지)
- 질문이 Claude/Anthropic API 관련이면 claude-api 스킬을 먼저 참고한다
