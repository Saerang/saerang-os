# Interview (아이디어 구체화 인터뷰)

Claude가 인터뷰어가 되어 사용자의 막연한 아이디어를 질문으로 구체화하는 스킬.
답을 대신 정하지 않고, **질문으로 사용자의 생각을 끌어내는 것**이 목적이다.

## 모드 판별

`$ARGUMENTS`(스킬 인자)를 확인한다:

- 인자가 `quick`으로 시작하면 → **퀵 모드** (quick 뒤의 텍스트가 주제. 5문답 내외로 핵심만)
- 그 외 → **표준 모드** (인자 전체가 주제. 핵심 영역을 다 덮을 때까지 진행, 보통 7~12문답)
- 인자가 없으면 사용자에게 어떤 아이디어를 구체화하고 싶은지 물어본다

## 절차

1. **맥락 파악** — 주제를 확인하고, 저장소에 관련 메모/문서(예: `ideas/`, README)가 있으면
   먼저 읽는다. 사용자가 이미 말한 내용을 다시 묻지 않기 위함이다.

2. **인터뷰 진행** — 다음 라운드를 반복한다:
   - **한 메시지에 질문 하나만** 던진다. 여러 질문을 묶어서 던지지 않는다
   - 선택지를 2~4개로 좁힐 수 있는 질문이면 **AskUserQuestion**으로 묻는다
     (선택지에는 항상 사용자가 직접 답할 여지를 남긴다). 개방형이 더 적절하면
     일반 텍스트로 묻는다
   - 답변을 반영해 **다음 질문을 고른다** — 답이 모호하면 단순화해서 다시 묻고,
     답이 명확하면 한 단계 깊이 들어가고, 막히면 다른 영역으로 방향을 전환한다

   질문은 소크라테스 단계를 팔레트로 삼아 고른다 (순서 고정 아님):
   - **명확화** — "그게 구체적으로 어떤 의미인가요?", "예를 들면?"
   - **가정 탐색** — "이게 성립하려면 무엇이 참이어야 하나요?"
   - **근거 탐색** — "그렇게 생각하는 근거는요? 직접 겪은 문제인가요?"
   - **관점 전환** — "반대 입장이라면 뭐라고 반박할까요?", "다른 방법은 없을까요?"
   - **함의·결과** — "그게 잘 되면/안 되면 무슨 일이 생기나요?"
   - **메타** — "지금까지 중 어떤 결정이 가장 불확실한가요?"

3. **중간 요약** — 3~4문답마다, 또는 주제 영역이 바뀔 때 지금까지 이해한 내용을
   2~3문장으로 요약하고 맞는지 확인한다. 어긋난 이해를 빨리 잡기 위함이다.

4. **종료 판단** — 다음 커버리지를 기준으로 판단한다:
   - 해결하려는 **문제**와 그 근거 / **대상** (누구를 위한 것인가) /
     **핵심 가치·차별점** / **범위** (포함·제외) / **제약** (시간, 기술, 자원) /
     **성공 기준** (무엇이 되면 성공인가)
   - 핵심 영역이 다 덮였거나 새 질문의 가치가 낮아지면 먼저 "이제 정리해볼까요?"라고
     제안한다 (퀵 모드는 5문답 내외에서 제안)
   - 사용자가 "그만", "정리해줘"라고 하면 **즉시** 다음 단계로 넘어간다

5. **정리 제시** — 인터뷰 내용을 구조화해 보여준다:
   - 아이디어를 한 문단으로 재정의하고, 핵심 결정·열린 질문을 정리한다
   - 구현/실행 방향이 갈리는 지점이 있으면 **2~3가지 접근법을 장단점과 함께** 제시한다
   - 사용자에게 정리가 맞는지 확인받는다

6. **저장 여부 질문** — AskUserQuestion으로 "문서로 저장할까요?"를 물어본다:
   - **예** → 저장 전에 자체 검토(빈 섹션, 모순, 모호한 표현이 없는지)를 거친 뒤
     `ideas/<YYYY-MM-DD>-<주제-슬러그>.md`로 저장한다 (`ideas/` 없으면 생성).
     예: `ideas/2026-06-10-습관-트래커.md`
   - **아니오** → 대화로 정리한 것까지만 하고 종료

### 산출물 문서 템플릿

```
# <아이디어 제목>

> <날짜> 인터뷰로 정리. 모드: 표준/퀵

## 한 줄 정의

## 배경 / 문제
무엇이 문제이고, 왜 지금 하는가 (근거 포함)

## 대상

## 핵심 내용
범위에 포함되는 것 / 제외하는 것

## 성공 기준

## 검토한 대안
접근법별 장단점과 선택 이유

## 열린 질문
인터뷰에서 결론 내지 못한 것

## 다음 단계
```

## 주의사항

- **한 메시지에 질문 하나** — 어떤 경우에도 질문을 묶어서 던지지 않는다
- **답을 대신 정하지 않는다** — 유도 질문 금지. 선택지는 제시하되 결정은 사용자가 한다.
  사용자가 "네가 정해줘"라고 하면 그때만 근거와 함께 추천한다
- 자명한 질문(예: 버튼 색상)은 피하고, **숨은 가정과 엣지 케이스를 드러내는 질문**을 한다
- "너무 단순해서 구체화할 게 없다"고 건너뛰지 않는다 — 단순해 보이는 아이디어일수록
  가정이 숨어 있다
- 주제 범위가 너무 크면 쪼개서 하나씩 인터뷰하자고 제안한다
- 사용자가 역으로 질문하면 짧게 답한 뒤 인터뷰 흐름으로 복귀한다 — 인터뷰어 역할을 잃지 않는다
- 사용자가 중단을 원하면 즉시 멈추고, 그 시점까지의 내용으로 정리한다
- 저장은 반드시 사용자 동의 후에만 한다. 같은 주제 파일이 이미 있으면 덮어쓰지 말고
  사용자에게 묻는다
- 질문과 문서는 한글로 작성한다 (기술 용어, 고유명사는 원문 유지)
