# Ralph (랄프 루프)

"똑똑한 한 방"이 아니라 **단순 무식한 반복이 이긴다** — 목표·지표를 받아 달성까지 이터레이션을
반복하는 루프. 컨텍스트가 차면 성능이 급락하므로, **매 이터레이션을 독립 컨텍스트 서브 에이전트로
돌리고**(작업 후 소멸), 진행 상황은 LLM 기억이 아니라 **파일(lessons.md)·git에 저장**해 다음
이터레이션이 이어받는다. 이 OS의 반복형·자율개선형 스킬의 토대다.

지표 작성·종료조건은 [references/metrics.md](references/metrics.md), 루브릭 판정은
[references/rubric-judge.md](references/rubric-judge.md)를 필요할 때 읽는다.

## 인자 / 목표 파싱

`$ARGUMENTS`에서 목표를 읽는다:

- 인자가 `--preset trigger <스킬이름>`이면 → **트리거 프리셋 모드** (아래 "트리거 프리셋" 참조).
  종료조건 4요소를 자동으로 채운다 — 묻지 않고 바로 시작 승인 게이트로 간다.
- 토큰 중 **실재하는 디렉터리 경로**가 있으면 대상 프로젝트 경로로 쓴다. 없으면 현재 작업
  디렉토리가 대상이다.
- 나머지는 **목표 서술**이다.
- 인자가 비어 있으면 무엇을 어느 지표까지 반복할지 물어본다.

**측정 가능한 종료조건이 없으면 시작하지 않는다.** 종료조건이 모호하면(예: "더 빠르게",
"품질 좋게") AskUserQuestion으로 **좋은 종료조건 4요소**를 확정한다 (한 번에 4개 이하):

1. **측정 가능한 종료 상태** — 무엇이 "달성"인가 (예: `npm test` 0 종료, E2E < 5분, 루브릭 전 차원 ≥ 4)
2. **증명 방법** — 어떻게 측정하나 (실행 명령 or 루브릭 차원·임계)
3. **지켜야 할 제약** — 건드리면 안 되는 것, 깨면 안 되는 것
4. **상한 N** — 최대 이터레이션 (예: 10). 상한 없는 무한 루프는 만들지 않는다

확정된 목표·지표·상한을 한 줄로 요약해 알리는 것이 **시작 승인 게이트**다 (develop 1단계와 동일).

## 절차

### 1. 셋업

- **goal-slug** — 목표 서술을 소문자 영문/숫자 kebab-case로 짧게 줄인다 (예: "E2E 시간 5분 이하" →
  `e2e-under-5min`). 같은 목표를 재시도할 때 브랜치·lessons가 이어지도록 결정적으로 짓고, 이미 있으면
  끝에 `-2` 등을 붙인다. 이하 `<slug>`로 줄여 쓴다.
- 작업은 **git worktree에 격리**해서 돌린다 (메인 작업물 보호 + 되돌릴 수 있게).
  `git worktree add .ralph/<slug>/wt -b ralph/<slug>`로 현재 HEAD 기준 격리
  작업 공간을 만든다. worker·judge의 작업·측정은 **모두 이 worktree 경로 안에서** 한다.
  - git 레포가 아니거나 worktree 생성이 실패하면, 그 사실을 알리고 "현재 브랜치에서
    직접 진행할지"를 사용자에게 물어본 뒤에만 진행한다 (이 경우 main/master면 새 브랜치 생성).
- 런 상태를 `.ralph/<slug>/`에 둔다 (worktree **밖**, 오케스트레이터가 관리):
  - `PROMPT.md` — 목표·종료조건·제약·증명 방법을 **한 번 정의하고 고정**한다 (매 이터레이션 주입)
  - `lessons.md` — 비워서 생성 (이터레이션마다 누적). worktree가 아니라 여기 둬서 라운드 간 유지
- `.ralph/`는 git 추적에서 빼도록 안내한다 (`.gitignore`에 없으면 추가 제안).
- **baseline 측정** — 루프 시작 전에 측정을 1회 돌려 시작 지표값을 잡는다 (`best`의 출발점이자
  개선/정체 판정의 기준). 정량/프리셋이면 명령·skill-tester를 실제 실행해 숫자를 확인한다.
  측정이 숫자로 안 나오면(명령 실패 등) **루프를 시작하지 않고** 그 사실을 보고한다.
  baseline이 이미 목표를 만족하면 할 일이 없음을 알리고 종료한다.

### 2. 이터레이션 루프 (pass / 3회 정체 / N회까지)

`라운드 = 1`, `best = baseline 측정값`, `정체 = 0`, `연속통과 = 0`으로 시작해 아래 정지 조건 중
하나가 될 때까지 반복한다 (`K` = 종료조건의 안정성 연속 통과 횟수, 기본 1 — 루브릭처럼 흔들리는
지표는 K>1로 받아 우연한 1회 통과를 막는다). 매 라운드:

1. **ralph-worker 서브 에이전트** (Agent 도구, `subagent_type: ralph-worker`)를 **매 라운드 새로**
   띄운다 — 다음을 전달한다:
   - `PROMPT.md`의 목표·제약, **대상 worktree 경로**(`.ralph/<slug>/wt`)
   - 현재 `lessons.md` 경로 (같은 실수 반복 회피용)
   - 직전 라운드 judge의 **실패 이유**(있으면)
   worker는 worktree 안에서 한 라운드 분량의 작업을 수행하고, 배운 것을 `lessons.md`에 append한 뒤
   짧은 요약만 반환한다. (독립 컨텍스트 → 작업 후 소멸. 메인은 요약만 받아 거의 안 참)
2. **측정** — worktree의 현재 상태를 측정해 **pass/fail + 실측 지표값**을 얻는다:
   - 기본: **ralph-judge 서브 에이전트** (Agent 도구, `subagent_type: ralph-judge`, 읽기 전용)에
     지표 유형·증명 방법·**대상 worktree 경로**를 전달 → pass/fail + 근거(정량 실측값/루브릭 차원 점수).
   - **트리거 프리셋 모드**: judge 대신 **skill-tester 서브 에이전트**(`subagent_type: skill-tester`)에
     worktree 안의 대상 `SKILL.md` 경로와 **고정된 태스크 세트**를 넘겨 트리거 pass rate(%)를 받는다.
     pass = 100%. (태스크 세트는 런 내내 고정 — 기준이 바뀌면 비교가 무의미하다.)
3. **개선/정체 판정** — 측정 실측값(정량 수치 / 루브릭 최저 차원 점수 / 체크리스트 충족 수 /
   트리거 pass rate)을 `best`와 비교한다:
   - 목표 방향으로 **개선**됐으면: `best` 갱신, `정체 = 0`, 이 worktree 상태를 유지(개선분 누적).
   - **개선 없으면**: `정체 += 1`. (연속 정체가 누적된다 — 개선이 나오면 0으로 리셋.)
   - 지표가 순수 이진(pass/fail만, 진척도 수치 없음)이라 개선을 잴 수 없으면 정체 추적은 건너뛰고
     pass 또는 N회 상한만으로 종료한다 (그 사실을 한 줄로 알린다).
   - **정량+루브릭 혼합 목표**면 두 지표를 각각 best로 추적하고, 아직 미달인 쪽을 다음 라운드 개선
     방향으로 삼는다 (둘 다 pass라야 ⓐ 달성 — metrics.md 합성 게이트).
4. 라운드 결과를 사용자에게 **한 줄로 보고**한다 (라운드 N: 실측 지표/점수, best, 정체 n/3, pass 여부).
5. **정지 조건** — 하나라도 충족하면 루프 종료:
   - ⓐ **달성**: `pass`면 `연속통과 += 1`, `fail`이면 `연속통과 = 0`. `연속통과 == K`가 되면 종료
     (K=1이면 첫 pass에 종료, K>1이면 연속 K회 통과해야 달성 — 안정성).
   - ⓑ **정체**: `정체 == 3` (3회 연속 개선 없음 → 수렴 정체).
   - ⓒ **상한**: `라운드 == N`.
   어느 것도 아니면 judge/skill-tester의 실패 이유를 다음 라운드 1번 worker에 넘기고 계속한다.

여러 도메인을 병렬로 돌리거나 라운드당 N개 worker를 띄워 최선을 채택하는 확장은
[references/parallel.md](references/parallel.md)에 설계만 있다 — v1은 순차 단일 worker다.

### 3. 종료 보고 (한글)

라운드별 지표 추이를 **표로** 보여주고(라운드 / 시도 요약 / 실측값 / best), 정지 이유를 명시한다:

- **달성**(pass): 라운드 수, 최종 지표/점수, baseline 대비 변화, `lessons.md` 핵심 교훈 요약, 변경 범위.
- **정체**(3회 연속 개선 없음) 또는 **N 소진**(미달성): 남은 격차(현재 지표 vs 목표), 시도한 것,
  막힌 원인, 선택지(상한 늘리기 / 목표·제약 조정 / 사람 개입 지점)를 보고한다 — **막혀도 보고가
  정상 출구**다 (make-skill·develop과 동일). 중도에 기준을 낮춰 억지로 pass시키지 않는다.

### 4. 사람 채택 게이트 (필수)

worktree에서 코드·파일을 실제로 바꿨으므로, worktree의 누적 변경을 `git diff`로 보여주고
AskUserQuestion으로 **"이 변경을 채택할까요?"**를 확인한다 (자동 머지·push하지 않는다):

- **채택** → `ralph/<slug>` 브랜치를 현재 작업 브랜치로 가져온다(`git merge`/`cherry-pick`,
  커밋은 commit 스킬 스타일 한글) 후 push·PR 여부를 이어서 묻는다
- **브랜치로 보존** → `ralph/<slug>` 브랜치를 그대로 둬 사람이 검토/PR하게 한다
- **폐기** → 브랜치를 버린다

처리 후 사용한 worktree를 `git worktree remove`로 정리한다(폐기 선택 시 브랜치도 삭제).
`.ralph/<slug>/lessons.md`는 다음 시도 참고용으로 남길지 묻는다.

## 트리거 프리셋 (`--preset trigger <스킬>`)

특정 스킬의 **트리거 정확도**를 자동으로 끌어올리는 내장 프리셋. 종료조건 4요소를 아래로 자동
세팅하므로 묻지 않고 바로 시작 승인 게이트로 간다:

1. **종료 상태**: 대상 스킬의 트리거 pass rate = 100%
2. **증명 방법**: `skill-tester` 서브 에이전트가 **고정된 태스크 세트**(should-trigger + near-miss +
   edge)로 채점한 pass rate(%). 측정은 judge 대신 skill-tester가 맡는다.
3. **제약**: 대상 스킬의 `SKILL.md`(주로 description)만 고친다. 기존에 잘 동작하던 트리거를
   깨지 않는다. 태스크 세트는 런 내내 고정한다.
4. **상한 N**: 기본 10.

매 라운드 worker는 worktree 안의 대상 `SKILL.md` description/본문을 한 번 개선하고, skill-tester가
재채점한다. pass rate가 개선되면 best 갱신, 아니면 정체 누적 — 100% 도달/3회 정체/N회에서 멈춘다.
태스크 세트가 없으면 시작 전에 worker가 대표 세트를 한 번 구성하고 그걸 고정한다.

## 주의사항

- **치팅 금지** — judge를 약화시키거나(테스트 skip·단언 제거·임계 낮추기) 산출물을 그럴듯하게만
  꾸며 통과시키는 것은 실패다. judge는 실제 실행/채점 기반이며, 종료조건은 시작 때 고정한다.
  AI가 의도를 배신하는 3패턴(치팅·포기·월권)을 막는 게 이 루프의 핵심 안전장치다.
- **상한 N 없는 무한 루프 금지** — 반드시 정지 3종(달성/정체 3회/상한 N) 중 하나에서 멈추고 보고한다.
- **worktree 격리 작업** — 변경은 격리된 worktree에서만 한다. main/master에 직접 커밋·push하지 않고,
  메인 브랜치로의 반영은 사람 채택 게이트를 거친다. 변경은 항상 되돌릴 수 있어야 한다.
- **종료조건은 시작 때 고정** — 루프 도중에 목표·임계·제약을 바꾸지 않는다 (바꾸려면 멈추고 새로 시작).
- **독립 컨텍스트 유지** — 이터레이션 작업을 메인에서 직접 하지 않는다. 매 라운드 새 worker에게
  위임하고 메인은 요약·판정만 받는다 (그래야 컨텍스트 오염을 피한다).
- headless 대신 서브 에이전트 방식을 쓴다 (며칠 돌릴 장기 루프가 아니면 서브 에이전트로 충분).
- 보고와 문서는 한글로 작성한다 (코드 식별자·고유명사는 원문 유지).
