# 얇은 적용 예시 — OS 스킬 자율 개선 루프

범용 `/ralph`를 이 OS 자신에게 겨눈 사례. "개선조차 AI가 한다"는 시스템 루프(글로벌 최적화)의
구체화로, 기존 자산(skill-stat · make-skill improve · code-review의 Gotcha)을 랄프 루프로 엮는다.
별도 스킬을 만들 필요 없이 `/ralph`에 아래 목표·지표를 주면 된다.

## 구성

- **목표(PROMPT)**: "스킬 `<X>`를 베스트 프랙티스 기준 점수가 통과할 때까지 개선한다."
- **worker = make-skill improve 절차**: ralph-worker가 라운드마다 `make-skill`의 improve 진단
  (frontmatter·description·500줄·참조 1단계·자유도·안티패턴)을 적용해 결함만 고친다. 잘 동작하던
  내용은 보존.
- **judge = skill-tester 점수**: ralph-judge가 정량 유형으로, 대표 태스크 세트에 대한 트리거
  정확도 + 태스크 pass 수를 측정한다 (skill-tester의 채점 형식 재사용).
- **종료조건**: "트리거 정확도 100% + 전 태스크 pass" (정량). 상한 N (예: 5).
- **lessons.md**: 어떤 결함을 고쳤고 무엇이 점수를 떨어뜨렸는지 누적 → 다음 라운드가 같은 결함을
  다시 만들지 않게.

## 대상 선정 — skill-stat 연동

어떤 스킬을 개선할지는 `/skill-stat`으로 고른다: 호출은 잦은데(자주 쓰임) 결함이 의심되는 스킬,
혹은 최근 실패가 관찰된 스킬을 우선순위로. 즉 **루프의 입력(무엇을 고칠지)** 을 skill-stat이 준다.

## 시스템 루프와의 관계

이건 단일 스킬 품질을 끌어올리는 **로컬 최적화**의 자동화다. 한 단계 위인 **글로벌 최적화**는
"세션 종료 → 회고 → 반복되는 문제를 OS DB에 기록 → 임계 넘으면 자율 개선" 같은 OS 전체 건강도
루프인데, 이는 `/ralph` 위에 별도로 설계할 후속 과제다. 지금은 `/ralph`가 그 토대를 제공한다.

## 안전장치

- 개선 대상 스킬·번들만 건드린다 (월권 금지).
- judge(skill-tester)를 약화시켜 통과시키지 않는다 — 태스크를 줄이거나 기대 동작을 낮추는 건 치팅.
- 최종 채택은 사람 게이트 (make-skill의 검수 게이트와 같은 철학).
