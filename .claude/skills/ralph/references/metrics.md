# 목표 지표 (metrics)

랄프 루프의 결과 품질은 **지표가 결정한다**. 앞의 셋(독립 컨텍스트·누적·병렬)이 인프라라면,
지표는 실제 결과를 좌우하는 가장 중요한 부분이다. 정량으로 잴 수 있으면 정량으로, 못 재는
정성 목표만 루브릭으로 간다.

## 좋은 종료조건 4요소

목표를 받으면 아래 넷이 다 채워졌는지 확인한다. 비면 사용자에게 묻는다.

1. **측정 가능한 종료 상태** — 무엇이 "달성"인지 이진/수치로 정의. ("더 빠르게" ✗ → "E2E < 5분" ✓)
2. **증명 방법** — 어떻게 측정하나. 실행 명령(`npm test가 0으로 종료`) 또는 루브릭 차원·임계.
3. **지켜야 할 제약** — 건드리면 안 되는 파일/동작, 깨면 안 되는 기존 동작.
4. **상한 N** — "또는 N턴 후 중단". 상한 없는 무한 루프는 만들지 않는다.

## 세 가지 지표 유형

### 정량 (quantitative) — 1순위
- 오류 0개 · 테스트 모두 PASS · 정확도 ≥ 90% · 시간 < 임계 등.
- 실제 명령 실행으로 객관 판정 → 비결정성 최소, 재현 가능. judge가 명령을 돌려 실측값을 본다.
- **가능하면 항상 이걸 우선한다.** 사용자의 "실행 기반 검증" 선호와 직결.

### 체크리스트 (checklist)
- 항목들의 충족 여부. "모두 통과" 또는 "n개 이상 통과".
- 정량화는 어렵지만 항목이 명확할 때.

### 루브릭 (rubric, LLM-as-judge)
- 정성 품질(UX·기획·카피 등)을 점수로 환산. 사람 또는 AI가 채점(AI면 LLM as a Judge).
- 작성·판정 규칙은 [rubric-judge.md](rubric-judge.md) 참조.
- 강점은 '정확성'이 아니라 **재현성** — 같은 답에 같은 점수를 일관되게 준다.

## 정량 + 루브릭 함께 쓰기

한 목표에 둘을 섞을 수 있다. 예: "기능은 테스트 통과(정량) + UX는 루브릭 전 차원 ≥ 4(루브릭)".
이때 judge는 **둘 다 pass여야 pass**로 판정한다 (게이트 합성). 정량으로 바닥(회귀 없음)을
지키고 루브릭으로 천장(품질)을 끌어올리는 조합이 안정적이다.
