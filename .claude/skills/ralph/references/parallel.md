# 병렬 베리언트 (확장 — v1 미구현)

v1 랄프 루프는 **순차 단일 worker**다. 한 라운드에 worker 하나가 작업하고 judge가 판정한다.
서브 에이전트를 백그라운드로 띄우면 **병렬**로 돌릴 수 있는데, 이 문서는 그 확장 설계만 적는다.
(구현 전이므로 SKILL.md 절차는 순차만 따른다.)

## 패턴 ① 라운드당 N개 worker → 최선 채택

같은 목표·같은 PROMPT를 N개의 worker에게 동시에 주고(각자 독립 컨텍스트), judge로 N개 결과를
채점해 **가장 좋은 결과 하나를 채택**한다.

```
같은 목표 → [worker A | worker B | worker C] (병렬) → judge 채점 → 최고 점수 채택 → lessons 누적
```

- 장점: 한 라운드의 기댓값이 올라간다 (다양한 시도 중 최선). 막다른 길에 덜 갇힌다.
- 비용: 토큰·복잡도 N배. judge 호출도 N배.
- 구현 골자: worker를 각각 다른 worktree(`isolation: worktree`)나 다른 브랜치에서 돌려 변경
  충돌을 막고, judge가 N개를 채점한 뒤 채택본만 메인 브랜치로 병합한다.

## 패턴 ② 여러 목표 병렬

독립적인 여러 목표(서로 다른 지표)를 각각 별도 랄프 루프로 동시에 돌린다. domain 스킬의
"여러 도메인 병렬 spawn"과 같은 관용구. 목표 간 의존이 없을 때만.

## v1에서 하지 않는 이유

- 검증을 단순하게 유지 — 순차 단일 worker가 정확히 동작하는지부터 확인한다.
- 병렬 worker는 파일 변경 충돌·채택 병합·토큰 비용 관리가 얽혀, 토대가 검증된 뒤 얹는 게 안전하다.
- 워크플로우 도구(`parallel`/`pipeline`, `isolation: worktree`)로 구현할 수 있으나, 이는 별도 확장
  작업으로 분리한다.
