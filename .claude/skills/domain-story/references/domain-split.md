# 도메인 분해 방법론 + 측정 루브릭

> domain-splitter가 따르고, 메인 오케스트레이터가 후보 분해안을 **채점·선택·검증**할 때 쓰는
> 단일 기준 문서. 목표: 코드 디렉터리를 **위키 페이지로 쪼개기 좋은 도메인 단위**로 잘 나눈다 —
> 너무 크지도(한 페이지에 다 못 담음) 작지도(파편화) 않게, 의미 경계와 RAG 청크 크기를 모두 맞춘다.

## 1. 하이브리드 분해 절차 (BC 1차 + 주제 2차)

**① 1차 — 바운디드 컨텍스트(BC)로 큰 덩어리를 가른다.**
`../domain/references/method-ddd.md`의 경계 신호를 쓴다:
- **유비쿼터스 언어 경계** — 같은 단어가 다른 의미면 경계 신호(가장 강함)
- **트랜잭션·일관성 경계** — 한 트랜잭션으로 묶이는 범위
- **모듈/패키지/디렉터리 구조** — 코드가 이미 그은 선(단, 코드 경계 ≠ 도메인 경계일 수 있음)
- **팀/소유·변경 속도·데이터 소유** — 함께 바뀌는 것끼리

**② 2차 — 큰 컨텍스트를 "독자가 한 번에 묻는 주제군"으로 쪼갠다.**
1차 컨텍스트가 위키 한 페이지에 담기에 너무 크면(엔티티 수십·규칙 수십·하위 기능 다수),
독자 질문 단위로 분할한다. **위키 한 페이지 = 한 주제군 = RAG가 한 번에 답하기 좋은 크기**.
단, 2차 분할이 BC 경계를 넘지 않게 한다(경계 누수 금지).

**③ 채널 모듈 전용 기능도 도메인 후보로 본다 (★누락 방지).**
도메인 신호로 **코어 모델 패키지(`*/model/<도메인>/`)만 보면 안 된다** — 코어 모델에 패키지가 없고
**채널/앱 모듈의 컨트롤러에만 사는 기능**(예: 소비자 로그인·소셜인증(OAuth)·멤버십·본인인증)이
통째로 누락된다. 채널 모듈(`*-api-*`, `*-app`, `*-web`)의 컨트롤러 패키지(`.../model/<기능>/`,
`.../api/model/<기능>/`)도 훑어, **모델 패키지가 없어도 컨트롤러군이 두꺼운 기능은 별도 도메인 후보**로 올린다.
(채널 모듈을 "액터"로 보는 것은 *전송 계층*에 한해서다 — 거기서만 사는 비즈니스 기능은 도메인이다.)

## 2. 측정 루브릭 (메인이 후보 분해안을 채점)

각 후보 분해안을 아래 7개 차원에서 0~100으로 채점한다. **최종 점수 = 최저 차원 점수(게이트)** —
한 차원이라도 치명적으로 낮으면 그 분해안은 탈락한다(전체 평균으로 약점을 가리지 않는다).

| 차원 | 무엇을 보나 | 낮은 예 |
| --- | --- | --- |
| 응집도 | 한 도메인이 한 비즈니스 능력에 집중하나 | 무관한 기능이 한 도메인에 섞임 |
| 결합도 | 도메인 간 의존이 최소인가 | 한 도메인 이해에 다른 도메인 절반이 필요 |
| 용어 경계 | 도메인 경계에서 용어 충돌이 해소되나 | 같은 용어가 두 도메인에서 다른 뜻 |
| 페이지 크기 | 위키 한 페이지/RAG 청크로 적정한가 | 한 도메인이 너무 방대 or 한 문단 |
| 질문 커버리지 | 예상 질문이 한 도메인 안에서 답되나 | 한 질문 답이 3개 도메인에 흩어짐 |
| 전체 균형 | 도메인 개수·크기 분산이 합리적인가 | 1개에 몰빵 or 20개로 파편화 |
| 커버리지 | 코드 전체가 어딘가에 속하나(고아 없음) — **코어 모델뿐 아니라 채널 모듈 전용 기능까지** | 소비자 로그인/인증이 어느 도메인에도 없음 |

## 3. 선택 후 검증 (한 번 더)

최고 점수 안을 고른 뒤 메인이 직접 확인한다:
- **고아 모듈(코어)**: 어느 도메인에도 안 들어간 코어 모델 패키지가 있나 → 있으면 누락
- **고아 기능(채널)**: **코어 모델엔 없지만 채널 모듈 컨트롤러에만 사는 기능**(소비자 로그인·소셜인증·
  멤버십·본인인증 등)이 어느 도메인에도 안 잡혔나 → 잡혔는지 컨트롤러 패키지 목록으로 교차 확인
- **경계 누수**: 한 모듈이 두 도메인에 걸쳐 있나 → 경계를 다시 긋거나 소유를 정한다
- **과대/과소**: 1~2개로 뭉쳤거나(덜 쪼갬) 20개로 흩어졌나(과하게 쪼갬)
문제가 있으면 splitter를 다른 시야로 다시 호출하거나 경계를 조정한다. **사람에게 묻지 않는다.**

## 4. splitter 반환 스키마

domain-splitter는 후보 분해안을 아래 형식으로 반환한다(메인이 채점할 수 있게):

```
## 후보 분해안 (시야: <DDD-경계 | 모듈-구조 | …>)
- 도메인: <한글 제목> (slug: <영문>)
  - 포함 경로/모듈: <디렉터리·패키지 목록> (근거)
  - 별칭: [<질문에 쓰일 말>, ...]
  - 한 줄 책임: <이 도메인이 담당하는 것>
  - 경계 근거: <왜 여기서 갈랐나 — 언어/트랜잭션/모듈/데이터 신호>
- 도메인: ...
## 자가 평가
- 응집도/결합도/용어경계/페이지크기/질문커버리지/균형/커버리지 각 코멘트
- [확인 필요] 경계가 모호한 지점
```

여러 시야가 필요하면 메인이 splitter를 복수 호출(예: DDD-경계 / 모듈-구조)해 후보를 모은 뒤
2절 루브릭으로 채점하고 최적안을 선택한다.
