# Make Skill (스킬 만드는 스킬)

Anthropic 공식 베스트 프랙티스를 이 OS 하우스 스타일에 박아, 새 스킬을 일관되게 높은 품질로
찍어내는 스킬. **핵심은 "만든 스킬이 실제로 정확히 동작하는가"를 실행 검증으로 보장하는 것**이다.

작성·점검 기준은 [references/best-practices.md](references/best-practices.md)에 있다 — 초안 작성과
자가 점검 때 필요한 절을 읽는다.

## 인자 / 모드 판별

`$ARGUMENTS`(스킬 인자)를 확인한다:

- 인자가 `improve`로 시작하거나, 인자가 기존 `.claude/skills/<이름>` 디렉터리를 가리키면
  → **improve 모드** (기존 스킬 진단·개선)
- 그 외 → **create 모드** (새 스킬 제작). 인자 전체를 "무엇을 하는 스킬인지"로 취급
- 인자가 없으면 → 무엇을 만들/고칠 스킬인지 물어본다

모드 판정 결과를 사용자에게 한 줄로 알린다.

## 절차 (create 모드)

1. **의도 캡처** — 다 정해지기 전엔 초안을 쓰지 않는다. AskUserQuestion으로 핵심만 묻는다
   (한 번에 4개 이하, 선택지엔 직접 입력 여지를 남긴다):
   - 무엇을 가능하게 하나 (스킬이 없을 때 매번 반복하는 일이 무엇인지)
   - **언제 트리거** — 사용자가 실제로 칠 말 (description의 근거가 된다)
   - 출력 형식 / 만들어 내는 결과물
   - 필요한 도구·의존성, 다뤄야 할 edge case
   - 번들 리소스가 필요한지 (참조 문서/스크립트/템플릿)

2. **초안 작성** — references/best-practices.md를 기준으로 SKILL.md를 쓴다:
   - `name`: 소문자·숫자·하이픈, 64자 이내, `claude`/`anthropic` 금지, vague 금지
   - `description`: **3인칭**, "무엇 + 언제 + 트리거 키워드", 1024자 이내 (한글)
   - 본문: **500줄 이하**, 자유도를 작업 위험도에 맞춤, "왜"를 설명, 한글(코드 식별자 원문)
   - 분량이 크면 `references/`·`scripts/`·`assets/`로 분할하되 **참조는 한 단계 깊이만**
   - 이 OS 하우스 스타일을 따른다: 인자/모드 판별 → 절차 → 주의사항 섹션, 사람 게이트, 한글 보고

3. **자가 점검** — 아래 인라인 체크리스트로 1차 거른다. 미달 항목은 고치고 다음으로 간다:
   - [ ] description 3인칭 + 트리거 키워드 구체적
   - [ ] name 규칙 충족 (소문자-하이픈, 예약어 없음)
   - [ ] 본문 500줄 이하, 참조 1단계
   - [ ] 시간민감 정보 없음, 용어 일관, 예시 구체적
   - [ ] 자유도 적절, 설명 과잉 없음
   (애매하면 references/best-practices.md의 "11. 출하 전 체크리스트" 전문을 본다)

4. **동작 검증 (skill-tester 서브 에이전트, 실행 기반)** — 품질의 핵심 단계다:
   - 먼저 **대표 태스크 세트**를 구성한다: should-trigger 프롬프트 + near-miss
     (비슷하지만 발동하면 안 되는) 프롬프트 + edge case. 각 태스크에 **기대 동작(assertion)** 명시
   - **skill-tester 서브 에이전트**(Agent 도구, subagent_type: `skill-tester`)에 초안 SKILL.md
     경로와 태스크 세트를 전달해 검증을 맡긴다 → 트리거 정확도 + 실제 실행 채점 보고를 받는다
   - 실패 항목은 **"스킬의 어느 문장이 원인인지"로 환원**해 고친 뒤 **재검증**한다
   - **전 태스크 pass + 트리거 정확**이 될 때까지 반복한다 (대충 통과시키지 않는다 — 치팅 금지)

5. **사람 검수 게이트 (저장 전 필수 — 생략 금지)** — 초안 전문과 검증 결과 요약을 보여주고,
   AskUserQuestion으로 **"이대로 저장할까요?"** 확인받는다. 수정 의견이 있으면 반영 후 다시 확인.

6. **저장** — `.claude/skills/<name>/SKILL.md`(+ 번들 파일)로 저장한다 (디렉터리 없으면 생성).

7. **커밋 제안** — AskUserQuestion으로 커밋 여부를 물어본다. 예 → commit 스킬 스타일의 한글로
   브랜치에서 커밋(예: `make-skill로 <이름> 스킬 추가`). 아니오 → 저장까지만.

8. **완료 보고** (한글) — 스킬 경로, 모드, 검증 결과 요약(트리거 정확도 / 태스크 pass 수),
   남은 개선 포인트, 새 세션에서 트리거 실증을 권장한다는 안내.

## 절차 (improve 모드)

1. 대상 스킬과 번들 파일을 읽는다
2. references/best-practices.md 대비 **진단** — frontmatter·description·500줄·참조 1단계·자유도·
   안티패턴·용어 일관성을 점검하고, 가능하면 실제 사용 양상(어디서 헤맸는지)도 반영한다
3. 변경안을 만든다 — **기존 내용(특히 잘 동작하던 부분)은 최대한 보존**, 결함만 고친다
4. 위 create 모드 4단계(동작 검증)를 동일하게 수행한다
5. 사람 검수 게이트 → 저장(기존 갱신) → 커밋 제안 → 완료 보고

## 주의사항

- **검수 없이 저장하지 않는다** — 어떤 경우에도 5단계(사람 검수 게이트)를 건너뛰지 않는다
- **검증을 대충 통과시키지 않는다** — 동작 검증은 실제 실행 기반이며, 전 태스크 통과가 출구다.
  테스트를 약화시키거나 기대 동작을 낮춰 통과시키는 것은 치팅이다
- 저장 전 반드시 검증: name 규칙 / description 3인칭 / 본문 500줄 이하 / 참조 1단계
- 시간민감 정보·Windows 경로(`\`)·voodoo 상수·선택지 과다 금지 (best-practices.md 10절)
- main/master에 직접 커밋·push하지 않는다 — 반드시 브랜치에서 작업
- improve 모드에서 기존의 잘 동작하던 내용을 근거 없이 삭제하지 않는다
- 더 엄격한 정량 벤치마크(트리거율 반복 측정, train/test 분할)가 필요하면 공식
  `skill-creator` 플러그인(`/plugin install skill-creator@anthropic-agent-skills`)을 안내한다
- 보고와 문서는 한글로 작성한다 (코드 식별자, 고유명사는 원문 유지)
