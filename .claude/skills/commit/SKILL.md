# Commit (한글 커밋 메시지)

변경사항을 분석해서 한글 커밋 메시지를 작성하고 커밋한 뒤, push 여부를 물어보는 스킬.
`$ARGUMENTS`(스킬 인자)가 있으면 커밋 메시지 작성 시 방향 힌트로 반영한다.

## 절차

1. **변경사항 파악** — 다음을 병렬로 실행:
   - `git status` — 변경/추가/삭제된 파일 확인
   - `git diff HEAD` — staged + unstaged 변경 내용 확인 (단, 추적되지 않은 신규 파일은 diff에 나오지 않으므로, `git status`에 뜬 신규 파일은 내용을 직접 읽어 파악한다)
   - `git log --oneline -5` — 최근 커밋 메시지 스타일 참고

2. **변경 내용 분석** — diff를 읽고 무엇이 왜 바뀌었는지 파악한다. 파일 나열이 아니라 변경의 의도와 효과를 요약한다.

3. **커밋 메시지 작성 (한글)**:
   - 첫 줄: 변경의 핵심을 한 문장으로 (50자 이내 권장). 예: `로그인 실패 시 재시도 로직 추가`
   - 변경이 여러 측면이면 본문에 한글 bullet으로 상세 내용 추가
   - 여러 줄 메시지는 heredoc으로 작성한다: `git commit -m "$(cat <<'EOF' ... EOF)"`
   - 메시지는 반드시 한글로 작성한다 (코드 식별자, 파일명 등 고유명사는 원문 유지)

4. **커밋 실행**:
   - 관련 파일을 `git add` (의도하지 않은 파일이 섞이지 않게 명시적으로 추가; `.env`, 비밀키, 대용량 빌드 산출물은 제외)
   - `git commit -m` 으로 커밋
   - 커밋 결과(`git log -1 --stat`)를 사용자에게 보여준다

5. **Push 여부 질문** — 커밋 완료 후 AskUserQuestion으로 "push 할까요?"를 물어본다:
   - **예** → `/push` 스킬의 절차대로 push 실행 (Skill 도구로 `push` 스킬 호출)
   - **아니오** → 커밋까지만 하고 종료

## 주의사항

- 변경사항이 없으면 커밋하지 말고 "커밋할 변경사항이 없습니다"라고 알린다
- pre-commit hook이 실패하면 원인을 보고하고, 임의로 `--no-verify`를 쓰지 않는다
- pre-commit hook이 파일을 자동 수정한 경우(포매터 등) 수정된 파일을 add하고 한 번만 재시도한다
- main/master 브랜치에서 작업 중이면 커밋 전에 사용자에게 알려준다
- 사용자가 요청하지 않은 무관한 변경 파일이 섞여 있으면 커밋에 포함하지 말고 따로 알린다
